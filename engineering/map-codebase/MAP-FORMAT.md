# CONTEXT-MAP.md Format

A `CONTEXT-MAP.md` at the repo root is the index for a multi-context (large) repo. It
tells agents **where to look** so they load only the relevant slice instead of scanning
the whole tree.

## Structure

```md
# Context Map

> Draft generated by /map-codebase. Boundaries and relationships are inferred from
> folder layout and import graphs — verify each with /grill-with-docs.

## Contexts

- **[Ordering](./src/ordering/CONTEXT.md)** — receives and tracks customer orders _(draft — verify)_
  - Code paths: `src/ordering/`, `src/api/orders/`
- **[Billing](./src/billing/CONTEXT.md)** — generates invoices and processes payments _(draft — verify)_
  - Code paths: `src/billing/`

## Relationships

- **Ordering → Fulfillment**: Ordering emits `OrderPlaced`; Fulfillment consumes it _(inferred — verify)_
- **Ordering ↔ Billing**: shared `CustomerId`, `Money` types _(inferred — verify)_
```

## Rules

- **Every context links to its `CONTEXT.md`** and lists the **code paths** it covers. The
  code paths are what let agents skip grep and read the right files directly — the core
  token saving.
- **Tag everything inferred.** Boundaries and relationships produced by automation are
  drafts; mark them `(draft — verify)` / `(inferred — verify)` until a human confirms via
  `/grill-with-docs`.
- **The map is an index, not a glossary.** No term definitions here — those live in each
  context's `CONTEXT.md`. No implementation details.
- **Keep relationships directional and concrete.** Name the event / type / call that
  creates the dependency, not just "depends on".

## Relationship to other docs

- `CONTEXT-MAP.md` (this file) — the index: which contexts exist, where they live, how
  they relate.
- `src/<context>/CONTEXT.md` — per-context glossary (filled by `/grill-with-docs`).
- `docs/adr/` — decisions; per-context decisions live in `src/<context>/docs/adr/`.

See the consumer rules in `docs/agents/domain.md` (seeded by `/hys-setup`) for how the
engineering skills read these files.
